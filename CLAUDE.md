# プロジェクト概要

クライアント社内システムのUI/UX刷新。
フロントエンドのみを製造し、クライアント既存APIを呼び出して情報を表示する。

## 技術スタック

- HTML / CSS / バニラJS（フレームワーク不使用）
- テスト: Playwright
- デプロイ: GitHub Pages
- バックエンド: GCP（クライアント既存API、詳細は別途確認）

## コーディングルール

### CSS
- テーマ値は必ずCSS変数（`--`プレフィックス）で管理する
- レイアウトと色は分離して管理する
- インラインスタイルは使わない

### JS
- モックデータと本番API呼び出しは差し替えやすい構造にする
- フレームワーク・ライブラリは原則使わない

### HTML
- セマンティックタグを使う

### 共通
- コメントは「なぜそうしているか」だけ書く（何をしているかは書かない）

## 自動化パイプライン

GitHubイシューを起点に、実装・テスト・デプロイまで自動で行う。
詳細は `.claude/skills/` 配下のSkillを参照。

```
GitHub Issue → issue-to-impl → test → ship
```

## テストと仕様書の連動（必須ルール）

過去に実装した挙動が新規実装で意図せず失われないことを保証するため、
UIの挙動を追加・変更したら **必ず** 次を1周回す:

1. 実装する
2. その挙動の E2E/ユニットテストを追加する。テスト名の先頭に `[SPEC-ID]` を付ける
3. `specs/<画面>.spec.md` に同じ SPEC-ID の仕様項目を追記/改定する
4. `npm test` を全緑にする（`spec-trace` で仕様⇔テストの一致を検証 → unit → e2e）

- SPEC-ID の採番・テンプレート・運用の詳細は `specs/README.md` を参照
- E2E は本番相当モックに依存せず、`test/fixtures/assets.fixture.js` を route 差し替えして使う
- 挙動を廃止するときは、仕様項目と対応テストを **両方** 同時に削除する
