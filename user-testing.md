## 0. 改訂メモ：agents.json と test.json の差分

テストケースは当初 agents.json（デモデータ）をもとに設計していたが、実際の API レスポンス構造は test.json であることが判明した。両者には以下の重大な差分がある。

| フィールド | agents.json（デモ） | test.json（実構造） | テストへの影響 |
|-----------|-------------------|-------------------|--------------|
| `department`（担当部門） | あり | **なし** | TC-04 を廃止 |
| `difficulty`（実装難度） | あり | **なし** | TC-06 を廃止 |
| `certified`（認定済み） | あり | **なし** | TC-03 を改訂 |
| `api`（API対応） | あり | **なし** | 関連テスト削除 |
| `pick`（ピック） | あり | **なし** | 関連テスト削除 |
| `rating` / `ratingCount` | あり | **なし** | ― |
| `capabilities[]` | なし | **あり**（9種） | 新TC追加 |
| `stats.download_count` | なし | **あり** | 新TC追加 |
| `owners[].ownership_type` | なし | **あり** (Director/Lead/Manager) | ― |
| `type.title`（カテゴリ） | カテゴリ一覧を別途定義 | 各アセットの`type`に内包 | 構造は同等 |

現行 agents.html の UI にある「実装難度」「担当部門」「認定済み」「API対応」「ピック」のフィルタはすべて実データに存在しないため、UI を test.json に合わせて更新した後のテストを想定して本レポートを改訂する。

## 1. テストの目的と成功指標

**測定したいこと**  
数千件規模に膨らむアセットカタログにおいて、ユーザーが目的のアセットを **3ステップ以内** に発見できるか。

**成功の定義**

| 指標 | 目標値 |
|------|--------|
| タスク完了率 | 80%以上 |
| 3ステップ以内達成率 | 70%以上 |
| タスク完了時間（中央値） | 60秒以内 |
| ユーザー主観評価（SEQ） | 5点以上 / 7点満点 |

> **ステップのカウント方法**: 「ページ遷移・フィルタ操作・検索実行・カード選択」のいずれかを1ステップとして計上する。初期ページ表示はカウントしない。

## 2. 実データに基づくサイト構造の整理

### test.json から導出できる検索・絞り込み軸

| 軸 | フィールド | 値の例 | UIでの想定実装 |
|----|---------|------|--------------|
| カテゴリ | `type.title` | 資料・文書作成、開発・コード支援 など9種 | サイドバーチップ |
| 機能タグ | `capabilities[].title` | Document Generation、Test Automation など9種 | サイドバーフィルタ または チップ |
| 並び順（人気） | `stats.view_count` | 0〜1198 | ソートドロップダウン |
| 並び順（利用実績） | `stats.download_count` | 0〜332 | ソートドロップダウン |
| 並び順（評価） | `stats.like_count` | 0〜n | ソートドロップダウン |
| 新着 | `last_updated_at` | 2025-10-15〜2026-06-19 | ソートドロップダウン / ナビ |
| キーワード | `title` / `overview`（HTML） / `owners[].full_name` | 全文 | 検索ボックス |

- API対応フラグ（api）
- プロデューサーズピック（pick）

## 3. テストケース一覧（改訂版）

### ケース設計の方針

- **業務シーン起点の指示文**を使う：「〇〇というアセットを探せ」ではなく「〇〇したいときに使えるものを見つけてほしい」
- 実データの検索・絞り込み軸（カテゴリ / capabilities / 並び順 / キーワード）のみを使うケースに限定する
- 弱点を炙り出す意図のあるケースは明示する

---

### TC-01: 業務文書を素早く仕上げたい（カテゴリ直行型）

**タスク指示文**  
> 「営業提案書や報告書のドラフトを書くのを手伝ってくれるアセットを見つけてください。」

**実データ上の正解**: `type.title = "資料・文書作成"` のアセット  
**想定パス**: カテゴリ「資料・文書作成」チップ選択 → カード選択（2ステップ）  
**難易度**: 低  
**測定ポイント**:
- カテゴリチップ「資料・文書作成」が直感的に目に入るか
- カテゴリ内に多数のアセットがある場合、デフォルト並び順（人気順）で上位のものが選ばれるか
- 「文書作成」「提案書」などのキーワード検索に走るユーザーがいないか（検索でもヒットするかの確認）

---

### TC-02: コードのバグを見つけたい（類似カテゴリでの迷い）

**タスク指示文**  
> 「書いたプログラムに潜むバグや品質問題を自動で発見してくれるアセットを探してください。」

**実データ上の正解**: `type.title = "テスト・品質保証"` または `"開発・コード支援"`  
**対応 capability**: `test-auto`（Test Automation）または `code-assist`（Code Assistance）  
**想定パス**: 検索「バグ」or「テスト」→ カード選択（2ステップ）  
**難易度**: 中（カテゴリが2つに分かれており迷いやすい）  
**測定ポイント**:
- 「テスト・品質保証」と「開発・コード支援」のどちらを選ぶか
- capabilities の「Test Automation」フィルタがあれば使うか
- overview のテキストが検索ヒットに十分な語彙を含んでいるか

---

### TC-03: 機能で絞り込みたい（capabilityフィルタ型）【新設】

**タスク指示文**  
> 「データを分析してグラフやサマリーを自動で出してくれるアセットが欲しいです。どのカテゴリにあるかはわかりません。」

**実データ上の正解**: `capabilities[].c_name = "data-analysis"` のアセット  
**想定パス**: capabilities フィルタ「Data Analysis / データ分析」選択 → カード選択（2ステップ）  
**代替パス**: カテゴリ「データ分析・診断」→ カード選択（2ステップ）  
**難易度**: 中  
**測定ポイント**:
- capabilities フィルタの存在に気づくか（カテゴリとの違いを理解できるか）
- 「カテゴリ」と「capabilities（機能タグ）」のどちらを先に使うかの行動パターン
- 「どのカテゴリかわからない」という条件で capabilities フィルタが代替手段として機能するか

> **UIへの要件フィードバック**: このTCで capabilities フィルタが発見されない場合、UI上での capabilities の露出度を上げる必要がある。

---

### TC-04: 実際によく使われているものを選びたい（ダウンロード数ソート型）【新設】

**タスク指示文**  
> 「社内で実際に多くの人がダウンロードして使っている、実績のある情報収集系のアセットを探してください。」

**実データ上の正解**: `type.title = "情報収集・調査"` かつ `stats.download_count` 上位のアセット  
**想定パス**: カテゴリ「情報収集・調査」選択 → 並び順「ダウンロード順（利用実績）」選択 → カード選択（3ステップ）  
**難易度**: 中  
**測定ポイント**:
- 「ダウンロード数で並べる」という操作が直感的に見つかるか
- 「人気順（ビュー）」と「ダウンロード順」の区別が理解できるか
- 3ステップ内に収まるか（カテゴリ → ソート変更 → カード選択）

---

### TC-05: 新しく追加されたものを確認したい（新着ナビ型）

**タスク指示文**  
> 「最近新しく追加されたアセットを確認したいです。」

**実データ上の正解**: `last_updated_at` 降順のアセット群  
**想定パス A**: グローバルナビ「新着」クリック → 一覧確認（1ステップ）  
**想定パス B**: 一覧ページで並び順「新着順」選択 → 一覧確認（1ステップ）  
**難易度**: 低  
**測定ポイント**:
- ナビの「新着」リンクに気づくか、並び順ドロップダウン経由かの経路分岐を観察する
- 両パスが同じ結果に到達するかを確認（一貫性の検証）
- 「先月以降」など期間指定ができないことに気づいたときの反応を記録する（今後の機能要件へのフィードバック）

---

### TC-06: 名前はうろ覚えで概要だけ覚えている（あいまい検索型・弱点炙り出し）

**タスク指示文**  
> 「以前同僚に教えてもらったアセットで、『自然言語でデータに問い合わせができる』と言っていたものを探してください。名前は覚えていません。」

**実データ上の正解**: `overview` に「自然言語でデータに問い合わせ」を含む `id:7004` など  
**想定パス**: 検索ボックスに「自然言語」または「データ 問い合わせ」と入力 → カード選択（2ステップ）  
**難易度**: 中〜高  
**測定ポイント**:
- overview（HTML文字列）がキーワード検索の対象になっているかどうか（**現状の実装次第でヒット率が変わる**）
- ヒットしなかった場合にカテゴリ探索に切り替えられるか
- このTCは検索実装の網羅性を測る「バグ炙り出し型」として位置づける

> **UIへの要件フィードバック**: overview の HTML タグを除去したテキストが検索対象に含まれていなければ、概要文中のキーワードは全てミスヒットになる。実装時に要確認。

---

### TC-07: オーナーから辿りたい（人名検索型）

**タスク指示文**  
> 「Yamamoto さんが作ったアセットがあると聞きました。探してください。」

**実データ上の正解**: `owners[].full_name` に "Yamamoto" を含むアセット（例: `id:7002` の Akiko Yamamoto）  
**想定パス**: 検索ボックスに「Yamamoto」と入力 → カード選択（2ステップ）  
**難易度**: 低〜中  
**測定ポイント**:
- オーナー名（`owners[].full_name`）が検索対象になっているかどうか
- ヒットした場合でも、一覧に「誰のアセットか」が表示されているか（カード上のオーナー情報の視認性）
- ヒットしなかった場合の行動（カテゴリ探索 or 諦め）

> **UIへの要件フィードバック**: 現状の検索が title / overview のみなら、オーナー名検索は機能しない。owners フィールドを検索対象に含めるかどうかは実装上の判断が必要。

## 5. テスト実施方法

### 5-A. モデレートあり対面テスト（推奨）

**対象人数**: 5〜8名  
**対象者プロファイル**:

| グループ | 人数 | 特性 |
|---------|------|------|
| ライトユーザー | 3名 | AIツールに不慣れな一般社員 |
| パワーユーザー | 2名 | AIツールを日常的に使うエンジニア・CoEメンバー |
| 管理職 | 2名 | 導入判断者として全体把握の視点を持つ |

**実施手順**

```
1. ウォームアップ (5分)
   - サイトを自由に閲覧してもらい、どんなサイトか説明してもらう
   - 第一印象を記録：「このサイトで何ができると思いますか？」

2. タスク実施 (30〜40分)
   - 優先度「高」の TC-01、TC-02、TC-03、TC-06 を必須実施
   - TC-04、TC-05、TC-07 は時間に応じて追加
   - シンクアラウドプロトコル：「今何を考えているか声に出してください」
   - モデレーターは誘導しない。3分で詰まった場合のみヒントを出す

3. 事後インタビュー (10分)
   - 「どこが一番困りましたか？」
   - 「カテゴリ以外の絞り込み（機能タグ）は使いましたか？どう感じましたか？」
   - 「もし1つ機能を追加できるとしたら何ですか？」
   - SEQ（Single Ease Question: 1〜7点）を各タスク直後に取得
```

**記録方法**
- 画面録画 + 発話録音（同意取得済みの前提）
- オブザーバーがステップ数・所要時間・詰まりポイントをリアルタイム記録

---

### 5-B. ファーストクリックテスト（事前スクリーニング）

本テスト前の簡易確認として画像2枚で実施する。

1. 「資料作成を助けてくれるアセットを探すには、まず何をクリックしますか？」  
   → カテゴリチップ / 検索ボックス / ナビ の分布を測る

2. 「機能（Document Generation など）でアセットを絞りたいとき、どこを操作しますか？」  
   → capabilities フィルタの発見性を事前確認する

## 7. 実データ構造から導出した UX リスク

### リスク1: capabilities フィルタの発見性（重大度：高）
カテゴリ（`type.title`）と capabilities（`capabilities[].title`）は別軸の概念だが、ユーザーには区別が難しい。カテゴリチップしか目に入らない場合、capabilities での絞り込みは使われない。  
**TC-03 でこのリスクを直接測定する。**

### リスク2: overview のキーワード検索カバレッジ（重大度：高）
overview フィールドは HTML 形式で格納されている。HTMLタグを除去したテキストが検索対象になっていない場合、ユーザーが概要文の語彙でキーワード検索してもヒットしない。数千件規模では致命的。  
**TC-06 でこのリスクを直接測定する。**

### リスク3: カテゴリとcapabilityの概念的重複（重大度：中）
「データ分析・診断」（カテゴリ）と「Data Analysis」（capability）のように、名称が類似した軸が複数存在する。ユーザーがどちらを使えばいいか混乱する可能性がある。  
**UI上での両者の役割説明が必要かどうかを TC-03 で確認する。**

### リスク4: オーナー名検索の非対応（重大度：中）
現在の検索実装が title / overview のみを対象としている場合、`owners[].full_name` は検索対象外になる。社内システムでは「あの人が作ったやつ」という人名起点の探索行動が頻繁に起きると予想される。  
**TC-07 でこのリスクを直接測定する。**

### リスク5: 大量件数での選択疲れ（重大度：中）
現在は80件のサンプルだが、数千件規模になったとき、カテゴリ単体では依然として数百件がヒットする。デフォルト並び順（人気順）が適切かどうか、また追加の絞り込み手段（capabilities、ダウンロード数）が自然に使われるかを TC-01 / TC-04 で確認する。

## 8. テスト後のアクション基準

| 結果 | アクション |
|------|----------|
| TC-06 でoverview検索がヒットしない | HTML除去テキストへの全文検索を最優先で実装 |
| TC-03 でcapabilityフィルタが発見されない（発見率 < 40%） | capabilities の UI 露出位置・ラベル・説明文を見直す |
| TC-02 でカテゴリ間の迷いが多発 | カテゴリ定義の見直し、またはcapabilityタグへの統合を検討 |
| 全体の3ステップ達成率 < 50% | 検索強化（owners / capabilities / overview 全フィールド対象）を緊急対応 |
| 全体の3ステップ達成率 70%以上 | 現行設計で数千件拡張フェーズへ進む |

---

*本レポートは test.json（実API構造、80件サンプル）および現行 agents.html の UI を分析した上で作成。agents.json のデモフィールド（department / difficulty / certified / api / pick）は実データに存在しないため、対応するテストケースを廃止・改訂した。*
