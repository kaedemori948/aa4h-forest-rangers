## 0. 改訂メモ：agents.json と test.json の差分

テストケースは当初 agents.json（デモデータ）をもとに設計していたが、実際の API レスポンス構造は test.json であることが判明した。両者には以下の重大な差分がある。

| フィールド | agents.json（デモ） | test.json（実構造） | テストへの影響 |
|-----------|-------------------|-------------------|--------------|
| `department`（担当部門） | あり | **なし** | TC-04 を廃止 |
| `difficulty`（実装難度） | あり | **なし** | TC-06 を廃止 |
| `certified`（認定済み） | あり | **なし** | TC-03 を改訂 |
| `api`（API対応） | あり | **なし** | 関連テスト削除 |
| `pick`（ピック） | あり | **なし** | 関連テスト削除 |
| `rating` / `ratingCount` | あり | **なし** | ― |
| `capabilities[]` | なし | **あり**（9種） | 新TC追加 |
| `stats.download_count` | なし | **あり** | 新TC追加 |
| `owners[].ownership_type` | なし | **あり** (Director/Lead/Manager) | ― |
| `type.title`（カテゴリ） | カテゴリ一覧を別途定義 | 各アセットの`type`に内包 | 構造は同等 |

現行 agents.html の UI にある「実装難度」「担当部門」「認定済み」「API対応」「ピック」のフィルタはすべて実データに存在しないため、UI を test.json に合わせて更新した後のテストを想定して本レポートを改訂する。

## 1. テストの目的と成功指標

**測定したいこと**  
数千件規模に膨らむアセットカタログにおいて、ユーザーが目的のアセットを **3ステップ以内** に発見できるか。

**成功の定義**

| 指標 | 目標値 |
|------|--------|
| タスク完了率 | 80%以上 |
| 3ステップ以内達成率 | 70%以上 |
| タスク完了時間（中央値） | 60秒以内 |
| ユーザー主観評価（SEQ） | 5点以上 / 7点満点 |

> **ステップのカウント方法**: 「ページ遷移・フィルタ操作・検索実行・カード選択」のいずれかを1ステップとして計上する。初期ページ表示はカウントしない。

## 3. テストケース一覧（改訂版）

### ケース設計の方針

- **業務シーン起点の指示文**を使う：「〇〇というアセットを探せ」ではなく「〇〇したいときに使えるものを見つけてほしい」
- 実データの検索・絞り込み軸（カテゴリ / capabilities / 並び順 / キーワード）のみを使うケースに限定する
- 弱点を炙り出す意図のあるケースは明示する

### TC-01: 実際によく使われているものを選びたい（DL実績フィルタ型）【更新】

**タスク指示文**  
> 「社内で実際に多くの人がダウンロードして使っている、実績のある情報収集系のアセットを探してください。」

**正解アセット**: **トレンドウォッチAI**（ID:8046）  
> カテゴリ「情報収集・調査」内でダウンロード数1位（DL:292）。DL実績順ソートまたは「DL実績 100件以上」フィルタで最上位に表示される。  
> ※ 同カテゴリ2位の「市場調査自動化AI」（ID:8001, DL:101）も正解として許容する。

**実データ上の正解**: `type.title = "情報収集・調査"` かつ `stats.download_count` 上位のアセット  
**想定パス A（ソート）**: カテゴリ「情報収集・調査」選択 → 並び順「利用実績順（DL）」選択 → カード選択（3ステップ）  
**想定パス B（フィルタ）**: カテゴリ「情報収集・調査」選択 → 「DL実績 50件以上」選択 → カード選択（3ステップ）  
**難易度**: 中  
**測定ポイント**:
- 「DL実績フィルタ（閾値）」と「ダウンロード数ソート」のどちらに先に気づくか、行動パターンを記録する
- 「人気順（ビュー）」「ダウンロード順（DL）」「DL閾値フィルタ」の3要素の違いをユーザーが区別できるか
- DL閾値フィルタ適用後に結果ゼロになったとき（閾値が高すぎる場合）にリセットできるか

### TC-02: 新しく追加されたものを確認したい（更新日フィルタ型）【更新】

**タスク指示文**  
> 「先月以降に新しく追加されたアセットを確認したいです。」

**正解アセット群**: 更新日「過去1ヶ月」フィルタで表示される37件が全て正解。  
> 代表例：**プレゼン資料テンプレ**（ID:7036）、**API利用ガイド**（ID:7038）、**ダッシュボード分析サンプル**（ID:7040）他（いずれも2026-06-19更新）  
> ※ このTCは特定のアセットへの到達よりも「フィルタ / ソート / ナビのどの経路をたどるか」の行動観察が目的。一覧が表示できた時点で完了と判定する。

**実データ上の正解**: `last_updated_at` が過去1ヶ月以内のアセット（2026-06-22 起点で37件）  
**想定パス A（フィルタ）**: 絞り込み「更新日 → 過去1ヶ月」選択 → 一覧確認（1ステップ）  
**想定パス B（ソート）**: 並び順「新着順」選択 → 一覧確認（1ステップ）  
**想定パス C（ナビ）**: グローバルナビ「新着」クリック → 一覧確認（1ステップ）  
**難易度**: 低  
**測定ポイント**:
- 3つのパスのうちどれを選ぶかを記録する（ナビ優先 / ソート優先 / フィルタ優先の分布）
- 「過去1ヶ月フィルタ」と「新着順ソート」の結果の違いをユーザーが理解できるか（フィルタは件数を絞る、ソートは順番を変えるだけ）
- 「過去1ヶ月」「過去3ヶ月」「今年」のプリセット粒度が適切かどうかの感想を取得する

### TC-03: 名前はうろ覚えで概要だけ覚えている（あいまい検索型・弱点炙り出し）

**タスク指示文**  
> 「以前同僚に教えてもらったアセットで、『自然言語でデータに問い合わせができる』と言っていたものを探してください。名前は覚えていません。」

**正解アセット**: **分析テンプレートセット**（ID:7004）  
> 概要に「自然言語でデータに問い合わせができ、グラフ・サマリーを自動生成します」と明記。キーワード「自然言語」で検索すると本アセットがヒットする（同語を含むアセットは他に「社内ナレッジ検索AI」(ID:8019)のみ）。  
> ※ テスト時に「自然言語」でヒットしない場合は **overview のHTMLタグ除去処理が機能していない** ことを示す重大バグとして記録する。

**実データ上の正解**: `overview` に「自然言語でデータに問い合わせ」を含む `id:7004`  
**想定パス**: 検索ボックスに「自然言語」または「データ 問い合わせ」と入力 → カード選択（2ステップ）  
**難易度**: 中〜高  
**測定ポイント**:
- overview（HTML文字列）がキーワード検索の対象になっているかどうか（**現状の実装次第でヒット率が変わる**）
- ヒットしなかった場合にカテゴリ探索に切り替えられるか
- このTCは検索実装の網羅性を測る「バグ炙り出し型」として位置づける

> **UIへの要件フィードバック**: overview の HTML タグを除去したテキストが検索対象に含まれていなければ、概要文中のキーワードは全てミスヒットになる。実装時に要確認。

### TC-04: ツール連携条件で探したい（タグフィルタ型）【新設】

**タスク指示文**  
> 「Excelと連携して使えるアセットを探したいのですが、どうすればいいですか？」

**正解アセット**: タグ「Excel対応」を持つ7件が全て正解。  
> 代表例：**ダッシュボード分析サンプル**（ID:7040, DL:128）、**売上トレンド分析AI**（ID:8000, DL:99）、**KPIダッシュボード自動化**（ID:8018, DL:94）  
> ※ タグ選択 / キーワード「Excel」検索どちらでたどり着いても正解。タグフィルタに気づかずキーワード検索のみを使ったケースは「タグ未活用」として記録する。

**実データ上の正解**: `tags[]` に `"Excel対応"` を含むアセット（7件）  
**想定パス**: タグ「Excel対応」ピル選択 → カード選択（2ステップ）  
**代替パス**: キーワード「Excel」検索 → カード選択（2ステップ）  
**難易度**: 中  
**測定ポイント**:
- 「タグ」フィルタの存在に気づくか（機能タグとタグの違いが混乱を生まないか）
- 「Excel対応」というタグラベルが直感的に理解できるか（CRM連携 / API連携 など他のタグも同様）
- タグとキーワード検索のどちらを先に試みるか
- タグラベルの網羅性に対する不満（「Teamsは？」「Power Automateは？」など）を記録する

> **UIへの要件フィードバック**: タグが空の場合（将来 tags[] を付与していないアセットが混在）、フィルタで意図しない抜け漏れが生じる。タグ付与の運用ガイドラインが必要。

### TC-05: 複数条件を組み合わせて安全なものを探したい（複合フィルタ型）【新設】

**タスク指示文**  
> 「最近追加されたアセットのうち、ある程度ダウンロード実績があって信頼できそうなものを探してください。カテゴリは問いません。」

**正解アセット**: 更新日「過去3ヶ月」かつ「DL実績 10件以上」で絞り込んだ結果に含まれる任意のアセットが正解。  
> 代表例：**商談準備アシスタント**（ID:8002, DL:332）、**E2Eテスト自動生成**（ID:8004, DL:280）、**セキュリティレビューAI**（ID:8039, DL:255）  
> ※ このTCは「複合フィルタ操作 + リセットボタンの認知」の確認が主目的。任意の1件にたどり着ければ完了とする。

**実データ上の正解**: `last_updated_at` 過去3ヶ月以内 かつ `stats.download_count >= 10` のアセット  
**想定パス**: 「更新日 → 過去3ヶ月」選択 → 「DL実績 → 10件以上」選択 → カード選択（3ステップ）  
**難易度**: 中〜高  
**測定ポイント**:
- 複数フィルターの同時使用（AND条件）が直感的に理解できるか
- フィルター適用順序が結果に関係ないことをユーザーが理解しているか
- 「リセット」ボタンの存在に気づき、一括解除できるか
- 「信頼性＝DL数」という判断軸をユーザーが自然に使うかどうか（DL閾値フィルタの有用性検証）

> **UIへの要件フィードバック**: 現在アクティブなフィルターが何件適用中かを示すカウンター（例：「2件のフィルター適用中」）があると、複合状態の把握が容易になる。

### TC-06: カテゴリをまたいで探したい（複数カテゴリ選択型）【新設】

**タスク指示文**  
> 「コード支援か品質テストに関係するアセットをまとめて見たいです。」

**正解アセット**: 「開発・コード支援」と「テスト・品質保証」の両カテゴリを選択した結果（計17件）に含まれる任意のアセットが正解。  
> 代表例：**API設計支援ツール**（ID:8026、開発・コード支援）、**バグ再現シナリオAI**（ID:8013、テスト・品質保証）、**UIテスト自動化ツール**（ID:8040、テスト・品質保証）  
> ※ このTCは「カテゴリの複数選択に気づくか」の確認が主目的。片方のカテゴリで目当てのアセットに到達できた場合も完了とする。

**実データ上の正解**: `type.title = "開発・コード支援"` または `"テスト・品質保証"` のアセット（計17件）  
**想定パス**: カテゴリ「開発・コード支援」選択 → さらに「テスト・品質保証」を追加選択 → カード選択（3ステップ）  
**難易度**: 高（複数選択の概念が伝わるかがポイント）  
**測定ポイント**:
- チップが複数選択できることにユーザーが気づくか（見た目の手がかりが十分か）
- 複数チップを選択した後、結果がOR（どちらか）で出ていることを理解できるか
- 「開発・コード支援 AND テスト・品質保証」を期待しているユーザーが混乱しないか（OR vs AND の認知のズレ）

> **UIへの要件フィードバック**: 複数選択時に「2カテゴリ選択中」のような状態表示があるとORロジックであることが伝わりやすい。

## 5. テスト実施方法

### 5-A. モデレートあり対面テスト（推奨）

**対象人数**: 5〜8名  
**対象者プロファイル**:

| グループ | 人数 | 特性 |
|---------|------|------|
| ライトユーザー | 3名 | AIツールに不慣れな一般社員 |
| パワーユーザー | 2名 | AIツールを日常的に使うエンジニア・CoEメンバー |
| 管理職 | 2名 | 導入判断者として全体把握の視点を持つ |

**実施手順**

```
1. ウォームアップ (5分)
   - サイトを自由に閲覧してもらい、どんなサイトか説明してもらう
   - 第一印象を記録：「このサイトで何ができると思いますか？」

2. タスク実施 (30〜40分)
   - 優先度「高」の TC-01、TC-02、TC-03、TC-06、TC-08、TC-09 を必須実施
   - TC-04、TC-05、TC-07、TC-10 は時間に応じて追加
   - シンクアラウドプロトコル：「今何を考えているか声に出してください」
   - モデレーターは誘導しない。3分で詰まった場合のみヒントを出す

3. 事後インタビュー (10分)
   - 「どこが一番困りましたか？」
   - 「カテゴリ以外の絞り込み（機能タグ・タグ・更新日・DL実績）は使いましたか？どう感じましたか？」
   - 「もし1つ機能を追加できるとしたら何ですか？」
   - SEQ（Single Ease Question: 1〜7点）を各タスク直後に取得
```

**記録方法**
- 画面録画 + 発話録音（同意取得済みの前提）
- オブザーバーがステップ数・所要時間・詰まりポイントをリアルタイム記録

---

### 5-B. ファーストクリックテスト（事前スクリーニング）

本テスト前の簡易確認として画像2枚で実施する。

1. 「資料作成を助けてくれるアセットを探すには、まず何をクリックしますか？」  
   → カテゴリチップ / 検索ボックス / ナビ の分布を測る

2. 「機能（Document Generation など）でアセットを絞りたいとき、どこを操作しますか？」  
   → capabilities フィルタの発見性を事前確認する

3. 「最近1ヶ月以内のアセットだけ見たいとき、どこを操作しますか？」  
   → 更新日フィルタ vs 新着ソート vs ナビの選択傾向を事前確認する

4. 「Excelと連携できるアセットを探すとき、まず何をしますか？」  
   → タグフィルタ vs キーワード検索の選択傾向を事前確認する

## 7. 実データ構造から導出した UX リスク

### リスク1: capabilities フィルタの発見性（重大度：高）
カテゴリ（`type.title`）と capabilities（`capabilities[].title`）は別軸の概念だが、ユーザーには区別が難しい。カテゴリチップしか目に入らない場合、capabilities での絞り込みは使われない。  
**TC-03 でこのリスクを直接測定する。**

### リスク2: overview のキーワード検索カバレッジ（重大度：高）
overview フィールドは HTML 形式で格納されている。HTMLタグを除去したテキストが検索対象になっていない場合、ユーザーが概要文の語彙でキーワード検索してもヒットしない。数千件規模では致命的。  
**TC-06 でこのリスクを直接測定する。**

### リスク3: カテゴリとcapabilityの概念的重複（重大度：中）
「データ分析・診断」（カテゴリ）と「Data Analysis」（capability）のように、名称が類似した軸が複数存在する。ユーザーがどちらを使えばいいか混乱する可能性がある。  
**UI上での両者の役割説明が必要かどうかを TC-03 で確認する。**

### リスク4: オーナー名検索の非対応（重大度：中）
現在の検索実装が title / overview のみを対象としている場合、`owners[].full_name` は検索対象外になる。社内システムでは「あの人が作ったやつ」という人名起点の探索行動が頻繁に起きると予想される。  
**TC-07 でこのリスクを直接測定する。**

### リスク5: 大量件数での選択疲れ（重大度：中）
現在は80件のサンプルだが、数千件規模になったとき、カテゴリ単体では依然として数百件がヒットする。デフォルト並び順（人気順）が適切かどうか、また追加の絞り込み手段（capabilities / タグ / DL閾値 / 更新日）が自然に使われるかを TC-01 / TC-04 / TC-09 で確認する。

### リスク6: タグフィルタとcapabilityの概念混同（重大度：中）【新規】
「機能タグ（capabilities）」と「タグ（tags）」の2種類のピルフィルターが縦に並んでいる。どちらも「タグ」的に見えるためユーザーが混乱し、片方しか使わない可能性がある。ラベルの表現やセクション見出しで役割の違いを明確にすべきかを **TC-08 / TC-03** で確認する。

> capabilities = 「何ができるか」（機能の種類）、tags = 「どんなツールと連携できるか / どんな用途か」（横断的な属性）という役割分担だが、ユーザーに伝わらなければUIの改善が必要。

### リスク7: フィルター複合時の状態把握（重大度：中）【新規】
複数のフィルターを組み合わせると、現在どのフィルターが効いているかが「リセットボタンの有無」だけでは把握しにくい。「3件のフィルター適用中」のようなサマリー表示がないと、ユーザーが自分の絞り込み状態を見失いやすい。特に更新日 × DL閾値 × タグの3重複合で問題が顕在化するかを **TC-09** で確認する。

### リスク8: OR条件の複数カテゴリ選択のメンタルモデルのズレ（重大度：低〜中）【新規】
複数カテゴリを選択するとOR結合で結果が表示される（「どちらかのカテゴリに属するもの全件」）。AND結合（「両方のカテゴリにまたがるもの」）を期待するユーザーは混乱する可能性がある。現実のデータではAND結合で絞りすぎることの方が問題になるが、認知の期待ズレは **TC-10** で測定する。

## 8. テスト後のアクション基準

| 結果 | アクション |
|------|----------|
| TC-06 でoverview検索がヒットしない | HTML除去テキストへの全文検索を最優先で実装 |
| TC-03 でcapabilityフィルタが発見されない（発見率 < 40%） | capabilities の UI 露出位置・ラベル・説明文を見直す |
| TC-08 でタグフィルタが発見されない（発見率 < 40%） | タグと capabilities のラベル・配置を統合または整理する |
| TC-08 で「このタグでは見つからない」という反応が多い | タグの種類・粒度・命名を見直す（ユーザー語彙に合わせる） |
| TC-09 で複合フィルタのリセット操作に詰まる | フィルター適用状態の可視化（チップ表示やバッジカウンター）を追加する |
| TC-10 でOR条件を誤解する参加者が過半数 | 複数選択時に「X カテゴリ選択中」の表示を追加し、OR結合であることを明示する |
| TC-02 でカテゴリ間の迷いが多発 | カテゴリ定義の見直し、またはcapabilityタグへの統合を検討 |
| TC-04 / TC-05 でフィルタとソートの区別がつかない | 「絞り込み」「並び替え」のセクション見出しと説明を強化する |
| 全体の3ステップ達成率 < 50% | 検索強化（owners / capabilities / tags / overview 全フィールド対象）を緊急対応 |
| 全体の3ステップ達成率 70%以上 | 現行設計で数千件拡張フェーズへ進む |
