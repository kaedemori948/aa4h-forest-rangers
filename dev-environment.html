01 — プロジェクト目的 / 概要

プロジェクト
概要

既存システムの機能はそのままに、操作性・視認性を改善し、業務効率の向上を目指す

目的

AIAgentサイトのUI/UXを刷新する。既存システムの機能はそのままに、操作性・視認性を改善し、業務効率の向上を目指す。

アーキテクチャ
  • —フロントエンドのみを実装し、クライアントの既存APIを呼び出して画面を表示する
  • —バックエンド・APIはクライアント側が管理（本プロジェクトでは触らない）
  • —モックデータと本番API呼び出しを差し替えやすい構造にし、開発中はモックで動作確認する
技術スタック
HTML CSS バニラJS CSS変数（--プレフィックス） セマンティックタグ フレームワーク不使用

07 — 仕様書① 仕様⇔テストの連動

仕様書とテストコードの1:1連動

specs/ 配下は挙動仕様のマスター。 テストコードと SPEC-ID で1:1連動させ、中長期の開発で過去挙動が意図せず失われていないことを保証する。

運用フロー（実装のたびに回す）
  • 1.実装する
  • 2.その挙動の E2E/ユニットテストを追加する（テスト名の先頭に `[SPEC-ID]` を付与）
  • 3.specs/<画面>.spec.md に同じ SPEC-ID の仕様項目を追記/改定する
  • 4.npm test を全緑にする（spec-trace で仕様⇔テストの一致を検証 → unit → e2e）

07 — 仕様書⑤ 要望フォーム 仕様一覧

specs/feedback.spec.md

要望フォーム（pages/feedback.html）

対応実装: pages/feedback.html インラインスクリプト ／ 全5項目

SPEC-ID 説明
FEEDBACK-001 入力に応じてIssueタイトルをライブプレビューする
FEEDBACK-002 必須未入力で送信するとアラートを出し、APIを呼ばない
FEEDBACK-003 必須を満たして送信すると Issue を作成し、成功バナーとリンクを表示する
FEEDBACK-004 送信が失敗するとエラーアラートを出し、ボタンを再有効化する
FEEDBACK-005 キーワードを Enter でチップ追加し、重複は無視する

SPEC-ID 採番ルール: SPEC-<画面>-<連番3桁>。 一度振ったIDは再利用・欠番の詰め直しをしない。挙動を廃止する場合は項目を削除せず「廃止」と明記し、対応テストも同時に削除する。 詳細ルールは specs/README.md を参照。
