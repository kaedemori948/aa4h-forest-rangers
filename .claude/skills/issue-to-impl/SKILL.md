# issue-to-impl

GitHubイシューを解析し、実装までを行う。確認なしで進める。

## トリガー
- `/issue-to-impl <Issue番号>`

## 手順

1. **イシュー取得**
   ```bash
   gh issue view <番号> --json number,title,body,labels,comments
   ```

2. **解析・計画**
   - 変更対象ファイルを特定する
   - 影響範囲（他ファイルへの波及）を確認する
   - 実装方針を1段落でコメントに残す（作業ログとして）

3. **実装**
   - CLAUDE.mdのコーディングルールに従う
   - モックデータと本番APIは差し替えやすい構造にする

4. **テスト（CLAUDE.md「テストと仕様書の連動」ルールに従う）**
   - UIの挙動を追加・変更した場合は必ず実施する。純粋なリファクタや非挙動変更（コメント修正等）のみの場合はスキップしてよい。
   - 対応するE2E/ユニットテストを追加・改定する。テスト名の先頭に `[SPEC-ID]` を付ける（採番・テンプレートは `specs/README.md` を参照）。
   - `specs/<画面>.spec.md` に同じ SPEC-ID の仕様項目を追記・改定する。挙動を廃止する場合は仕様項目とテストを両方削除する。
   - E2E は `test/fixtures/assets.fixture.js` を route 差し替えして使う（本番相当モックに依存しない）。
   - `npm test`（`test:trace` → `test:unit` → `test:e2e`）を実行し、全緑になるまで実装・テストを直す。
   - 全緑になるまで次のステップ（チェックリスト更新・PR作成）に進まない。

5. **チェックリスト更新（`dev-fix` ラベルの場合）**
   - issue本文がチェックボックス形式の修正案リストの場合、実装が完了した項目を `- [ ]` → `- [x]` に書き換える。
   - `gh issue edit <番号> --body "<書き換え後の本文全体>"` で反映する（本文全体を渡すため、既存の注釈・見出しは維持したまま該当行だけ変更する）。
   - 一部の修正案のみ実装した場合でも、実装できた項目だけチェックを入れ、未実装の項目はそのまま残す。

## 注意
- PRには実装に直接関係するファイル（HTML / CSS / JS / テスト / 仕様書）のみを含める。
- CSS変数を新たに追加する場合は `app.css` のルート変数に定義する
- 既存のCSSクラスを再利用できないか先に確認する
