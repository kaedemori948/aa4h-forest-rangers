# UI改善案 — Issue #14「[アセット要望] 営業提案書ドラフト生成AI」（サイクル2）

## 前提の整理

### 前回（サイクル1）のアプローチと結果

- 実装: `pages/agents.html` のインラインスクリプトで `window.ASSETS_DATA.assets` に
  架空アセット (id 9001) を push してカード表示
- 結果: **却下**（`proposal: revise`）。PRの却下理由テンプレートは未記入だが、
  実在しないアセットの捏造・本番API切替時に成立しない構造・データとビューの分離違反が
  構造的な問題として明らかであるため、本サイクルでは「データを足す」系のアプローチを全て候補から除外する。

### データ調査で判明した事実（今回の分析の核心）

要望者が求める機能は **既にカタログに存在する**:

| id | タイトル | カテゴリ | 内容 |
|----|---------|---------|------|
| 8007 | 提案書自動生成ツール | 資料・文書作成 | 顧客業種・課題・予算を入力し提案書を自動ドラフト（要望とほぼ同一） |
| 8020 | 提案資料パーソナライザー | 営業・顧客対応 | 顧客プロファイルから提案資料を自動カスタマイズ |
| 7044 | 営業提案テンプレ | 営業・顧客対応 | BtoB提案書のひな型＋記述ガイド |

にもかかわらず要望者（キーワード: `提案書` `営業` `ドキュメント生成`）が到達できなかった理由は
検索仕様にある:

1. **複数キーワードは全トークンAND** かつ検索対象が **タイトル/オーナー/概要のみ**。
   「営業 提案書 ドラフト」で検索すると **0件** になる
2. **タグ・機能（capabilities）・カテゴリが検索対象外**。
   「ドキュメント生成」は 8007 の機能「文書生成 / Document Generation」にヒットしない（語彙のズレ）
3. **検索時も人気順ソート**のため、「営業」(75件) のような広いキーワードでは
   タイトル完全一致のアセットが埋もれる（「件数は多かったが目的のものがなかった」の正体）

`data/` は変更禁止のため、この語彙・データ形状起因の問題を全て表示側（JS）で吸収する。

## フェーズ2: ビジュアル設計計画

本案はロジックのみの改善（検索マッチング・並び順）であり、新規UI要素・スタイル変更を

## 実装仕様（案A詳細）

`assets/app.js` `renderList()` の `compute()` を拡張:

1. **検索対象フィールド拡張**: タイトル・オーナー・概要（従来）＋タグ・機能・カテゴリ名
2. **同義語展開**: 小規模な同義語グループ辞書（表示側で保持）でトークンを展開
   - 提案書 / 提案資料 / プロポーザル
   - ドキュメント生成 / 文書生成 / ドキュメント作成 / 文書作成
   - ドキュメント / 文書 / 資料
   - 生成 / 作成 / ドラフト
   - 営業 / セールス
3. **関連度スコアリング**: フィールド重み（タイトル4 > タグ3 > 機能2 = カテゴリ2 > 概要1.5 > オーナー1）。
   一致トークン数の多い順 → スコア順 → 選択中の並び順、で表示。
   1トークンでも一致すれば結果に含める（AND全滅による0件を防ぐ）。検索語なしの挙動は不変
